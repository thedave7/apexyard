apexstack

Where projects get forged.

The multi-project ops repo where your projects reference each other, learn from shared experience, and ship production-ready under a strict SDLC. You don't add apexstack to a project — projects get forged inside it. One ops repo. Every product. Shared memory. Strict gates. Production-ready MVPs. Claude Code is the default driver, but the rules, hooks, and templates are plain markdown and shell. Swap the AI. Keep the forge.

★ Star on GitHub · or install it now ↓
$ git clone https://github.com/me2resh/apexstack.git .apexstack

Run from your ops repo — the one you'll use to govern every product. Full walkthrough in install below. No magic. No build. No SaaS. Prefer a single global copy shared across every ops repo? That's also supported.

02 / LAYERS

Five layers, one stack

ApexStack is built around managing a portfolio of projects, not a single repo. Five layers stack together: roles, workflows + templates, hooks + rules, agents + skills, and the workspace itself — a registry of every repo under your org's care. Single-project mode still works for the simple case, but the default is multi-project.

01 — passive · markdown

Roles

19 software roles across 5 departments. Each role has identity, responsibilities, capabilities, boundaries, interfaces, and a checklist. Tell Claude "act as the QA Engineer" and it picks up the full mental model.

  • engineering/7 roles — Head, TL, BE, FE, QA, Platform, SRE
  • product/3 roles — Head, PM, Analyst
  • design/3 roles — Head, UI, UX
  • security/3 roles — Head, Auditor, Pen Tester
  • data/3 roles — Head, Analyst, Engineer
02 — passive · markdown

Workflows + templates

Three workflow processes (full SDLC, code review, deployment) and four document templates. Drop them into your docs/, ship faster.

  • sdlc.md7-phase lifecycle with quality gates
  • code-review.mdstructured author + reviewer process
  • deployment.mdstaging → prod with rollback
  • prd.mdproduct requirements doc
  • agdr.mdAgent Decision Record (AI-specific)
03 — runnable · executable NEW

Hooks + rules

Six shell hooks block dangerous git operations and enforce standards on every Bash tool call. Six modular rule files import into your CLAUDE.md via @-imports so Claude inherits the standards in every conversation.

  • hooks/git add -A · main push · secrets · branch / PR format
  • rules/AgDR triggers, code standards, PR quality, gates
  • settings.jsonwires hooks to PreToolUse events
04 — runnable · invokable NEW

Agents + skills

Five sub-agents with their own model, prompt, and tool access. Thirteen slash commands users type into Claude Code. Both designed to be invoked from any conversation in any of your projects.

  • agents/Rex · Shield · Guardian · PR Manager · Ticket Manager
  • skills//decide · /code-review · /security-review · /audit-deps · /write-spec
  • skills//idea · /handover · /projects · /inbox · /status · /tasks · /roadmap · /stakeholder-update
05 — workspace · multi-project by default NEW

A portfolio, not a single repo

ApexStack treats your engineering org as a portfolio of projects, not a single codebase. The default install creates an ops repo with an apexstack.projects.yaml registry listing every repo you manage — and every skill that operates on tickets, PRs, issues, status, or roadmap (/projects, /inbox, /status, /tasks, /handover) aggregates across the registry by default. If you only have one repo, ApexStack still works — set mode: single-project in onboarding.yaml and the same skills scope to the current repo. workspace/<name>/ holds live local clones; projects/<name>/ holds the long-lived committed docs.

  • onboarding.yamlapexstack.mode: multi-project (default) or single-project
  • apexstack.projects.yamlthe portfolio registry — name, repo, workspace path, docs path, status, applicable roles
  • workspace/<name>/live local clones, gitignored — your daily working copies
  • projects/<name>/committed per-project docs (README, roadmap, AgDRs, handover assessments, stakeholder updates)
  • docs/multi-project.mdfull guide — registry schema, daily workflow, migration from single-project

04 / PIPELINES

Drop-in CI workflows

Seven GitHub Actions workflows live at golden-paths/pipelines/. Copy whichever you want into your project's .github/workflows/ — no boilerplate to write, no configuration to wrestle with.

ci.yml
Combined pipeline — code quality + security + dependencies in one file. Start here.
Rex · Shield · Guardian
code-quality.yml
TypeScript check, ESLint, Prettier, tests with coverage, build verification. Posts a summary comment on every PR.
Rex
security.yml
Semgrep SAST (OWASP Top 10), npm audit, TruffleHog secrets detection, optional CodeQL. Fails on critical or high severity.
Shield
dependency-audit.yml
Weekly vulnerability scan (Mondays 09:00 UTC) plus on every package.json change. Auto-creates a security issue when something critical lands.
Guardian
pr-title-check.yml
Enforces a ticket ID in every PR title. Default: GitHub Issues (#58). Also accepts any uppercase tracker prefix.
governance
review-check.yml
Blocks merge if Rex hasn't reviewed the latest commit on the PR. Compares review SHAs against HEAD.
Rex (verification)
seo-check.yml
SEO scoring for content files — H1, meta description, length, headings, alt text, internal links. For docs and blog sites.

05 / INSTALL

Set up your portfolio in six steps

ApexStack defaults to multi-project mode — one ops repo, one registry, every managed project listed inside it. Single-project mode (one repo only) is a one-line opt-out at the end if that's what you actually need.

01 — clone

Clone ApexStack into your ops repo

The ops repo is where ApexStack lives — typically a dedicated repo like your-org/ops, or an existing internal-tools repo. ApexStack itself sits at .apexstack/ inside it.

# in your ops repo (or create one)
git clone https://github.com/me2resh/apexstack.git \
  .apexstack
02 — expose .claude/

Symlink the runnable layer

Claude Code looks for hooks, agents, skills, and settings.json at .claude/. Symlink so updates in apexstack stay in sync.

# in your ops repo root
ln -s .apexstack/.claude .claude
03 — configure

Fill in onboarding.yaml

Tell ApexStack about your team — roles, tech stack, quality bar. The default mode is multi-project; leave it as is unless you only have one repo.

# defaults are sane — edit in place
$EDITOR .apexstack/onboarding.yaml
04 — list your projects

Create the registry

Copy the example registry and list every repo you want under ApexStack management. One YAML entry per project — name, repo, status, applicable roles. The cross-project skills (/projects, /inbox, /status, /tasks) read from this file.

# in your ops repo root
cp .apexstack/apexstack.projects.yaml.example \
   apexstack.projects.yaml
$EDITOR apexstack.projects.yaml  # list your repos
05 — wire CLAUDE.md

Point your CLAUDE.md at the stack

A single @-import line in your ops repo's CLAUDE.md pulls in the entire stack — rules, role definitions, workflows, templates, and the multi-project skill set.

# in your ops repo CLAUDE.md
@.apexstack/CLAUDE.md
06 — start working

Try a portfolio command

In any Claude Code session inside the ops repo, the cross-project skills are immediately available. Hooks fire on every Bash call. Done.

# in claude code, from the ops repo
/projects          # list every managed project + status
/inbox             # PRs, issues, comments needing your eyes
/status            # git + CI snapshot per project
/decide migrate billing to Stripe
opt-out — single-project mode

If you actually have just one repo

Skip step 04 entirely (no registry needed) and set apexstack.mode: single-project in onboarding.yaml. ApexStack lives directly inside that one repo, not a separate ops repo. The same skills scope to the current repo, the roadmap lives at ROADMAP.md, ideas at IDEAS.md. Everything else is identical.

# in onboarding.yaml
apexstack:
  mode: single-project
alternative — global install

If you run several ops repos

The six-step default clones apexstack into each ops repo at .apexstack/. That's the right call for most people — upgrades are per-repo, nothing is shared across machines. If you run several ops repos (a founder with a couple of orgs, a consultant with multiple clients), clone apexstack once globally and reference it from every ops repo. One place to upgrade, one place to patch.

Tradeoff: the symlinks point at an absolute path in your home directory, so moving ~/.apexstack/ breaks the link. Not a good fit if you sync dotfiles across machines with different home paths.

# one time, globally
git clone https://github.com/me2resh/apexstack.git ~/.apexstack

# in each ops repo (replaces steps 01 and 02 of the default flow)
ln -s ~/.apexstack/.claude .claude
echo '@~/.apexstack/CLAUDE.md' >> CLAUDE.md

# steps 03, 04, 05, 06 are the same as the default flow
06 / SCOPE

Opinionated by design

Every reusable framework has to draw a line between "this is what we do" and "this is yours to figure out". ApexStack draws the line where it makes the most difference for engineering teams.

+ what apexstack does

  • Enforces ticket-first development via GitHub Issues per project
  • Forces structured decisions via /decide → AgDR audit trail
  • Blocks dangerous git operations (add -A, push to main, secrets in commits)
  • Ships an opinionated 6-gate workflow from PRD to QA
  • Mandates a glossary in every PR — knowledge stays in the team
  • Defaults to TypeScript, DDD, vitest, GitHub Actions — but stays editable
  • Treats hooks, agents, and slash commands as a single coherent surface

× what apexstack does NOT do

  • Replace your team — humans still write code, ship product, talk to customers
  • Lock you into a SaaS — every file is markdown or shell, fork freely
  • Hide its opinions — read the rules, disagree, edit them
  • Pretend to be language-agnostic — TypeScript-leaning by default
  • Ship a UI — this is a stack of files, not an app
  • Impose a central tracker — issues live in each project's own GitHub repo
  • Compete with Claude Code's built-ins — it extends them

Stop reinventing
the same processes.

ApexStack is one git clone, one symlink, and one config file away. It's MIT licensed, plain markdown, and edits are encouraged. If you change your mind, you delete the directory and you're back where you started.
